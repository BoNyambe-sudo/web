# Plan: Fix BlogComment Recursive Actions

## Problem Analysis

1. **`useCreateComment` onSuccess cache update mismatch**: When creating a reply, it updates query key `["replies", parentId]` but the actual query key is `["replies", blogId, commentId, params]` (missing blogId and params). This causes replies not to appear immediately.

2. **`useLikeComment` and `useDislikeComment` don't invalidate replies cache**: They only invalidate `["comments", blogId]`, but replies are stored under `["replies", blogId, commentId, params]`. So likes/dislikes on replies don't update the UI.

3. **`onReply` callback called prematurely**: Called outside `onSuccess`, may fire before the reply is actually created.

4. **Potential infinite recursion**: The component calls itself recursively without a depth limit, but this is intentional for nested replies. The real issue is that the mutations don't properly update the cache for nested levels.

## Solution

### 1. Fix `useCreateComment` (useBlogServer.tsx)
- Update `onSuccess` to use the correct query key: `["replies", variables.blogId, variables.parentId]`
- Also invalidate the parent comment's like/dislike state by including the parent comment ID
- Consider also invalidating the main comments query if needed

### 2. Fix `useLikeComment` and `useDislikeComment` (useBlogServer.tsx)
- In `onSuccess`, invalidate both:
  - Main comments query: `["comments", variables.blogId]`
  - Replies query for that specific comment: `["replies", variables.blogId, variables.commentId]`
- This ensures likes/dislikes work at any nesting level

### 3. Fix `useUpdateComment` (useBlogServer.tsx)
- Also update replies cache correctly (currently only checks `["replies", variables.commentId]` which is missing blogId and params)
- Should invalidate `["replies", variables.blogId, variables.commentId]`

### 4. Fix BlogComment.tsx
- Move `onReply?.(comment)` call inside the `onSuccess` callback of `createReply` mutation
- Remove duplicate `setReplyText("")` after `createReply` call (it's already in onSuccess)
- Optionally add a max depth prop to prevent excessive recursion (but likely not needed)

### 5. Verify `useDeleteComment` and other mutations
- Ensure they also invalidate the correct reply query keys for consistency

## Implementation Order

1. Fix `useCreateComment` - correct query key for replies
2. Fix `useLikeComment` - add replies cache invalidation
3. Fix `useDislikeComment` - add replies cache invalidation
4. Fix `useUpdateComment` - correct query key for replies
5. Fix `BlogComment.tsx` - move onReply callback and clean up state management
